Revisión post-implementación de la pantalla `/memory` Liminal. Consolida tres investigaciones paralelas sobre el repo `daimon` (backend Go) y `daimon-frontend` (React/Vite).

**Ejes cubiertos**
1. ¿Qué tanto podemos alimentar la nueva pantalla de Memory con data real?

## 1. Memory / RAG — qué es poblable con data real

**Endpoints** — `internal/web/server.go:232-234`
- `GET /api/memory?q=&scope=&limit=` → búsqueda FTS5
- `POST /api/memory` → crear

**Curator (autoaprendizaje)** — `internal/agent/curator.go`
- Corre después de CADA respuesta no-vacía (`internal/agent/loop.go:291`).
- Clasifica con LLM (`importance`, `type`, `topic`, `title`).
- Filtros de bajo ruido: respuestas cortas, refusals, fillers → drop.
- Dedup por cosine similarity (embeddings) o Jaccard (fallback).
- Conclusión: **"cosas que Daimon aprendió por observación" ya existe** — no hay que inventarlo.

**Extensiones mínimas para cerrar el mock** (no son prerrequisitos, el panel ya funciona sin ellas):
1. Agregar columna `cluster TEXT NULL` a `memory_entries`.
2. Enriquecer el Curator para clasificar `cluster` con un prompt LLM corto (5 categorías fijas: identity/preferences/projects/relationships/technical).
3. Opcional: persistir `source.msg` (ID de mensaje) además de `source.conv`.

### 1.4 Resumen ejecutivo — Memory/RAG

## 2. Chat — lag de tipeo + jank

**Síntoma**: varios segundos entre keystroke y aparición de letras; chat se siente "tosco".

### 2.1 Causa primaria

**Cada keystroke re-renderiza TODO el árbol del chat** — historial completo con markdown, code blocks y reasoning.

**Evidencia en cadena**
- `src/components/liminal/LiminalInput.tsx:77` — `onChange={(e) => onChange(e.target.value)}` dispara `setInput` en `ChatPage`.
- `ChatPage.tsx` — `input` y `messages` viven en el mismo componente. Cualquier `setInput` re-renderiza `<ChatPage>`, lo que implica re-renderizar la lista completa de `LiminalUserMsg` + `LiminalAssistantMsg`.
- `LiminalAssistantMsg.tsx:28` — **no está envuelto en `React.memo`**. Se re-renderiza aunque sus props no cambien.
- `LiminalMd.tsx:180` — **tampoco está memoizado**. `ReactMarkdown` + `remark-gfm` re-parsean el markdown en cada render.
- `LiminalCode.tsx:197-198` — tiene `useMemo` para la tokenización, pero el componente padre se re-renderiza igual, así que el work de shallow-equal prop check también ocurre.
- **React.StrictMode** (`src/main.tsx:7`) duplica renders en dev — amplifica el lag 2×.

### 2.2 Amplificador — auto-resize del textarea

`LiminalInput.tsx:45-50`

```ts
useEffect(() => {
  const el = textareaRef.current
  if (!el) return
  el.style.height = 'auto'
  el.style.height = `${Math.min(el.scrollHeight, 180)}px`
}, [value])
```

Lee `scrollHeight` justo después de cambiar `height = 'auto'`. Es un patrón clásico de **layout thrashing**: obliga al browser a recalcular layout síncronamente. Por sí solo es ~1-3 ms; el problema es que ocurre EN MEDIO del re-render del chat histórico, no antes ni después.

### 2.3 Amplificador — stream de reasoning compite con el input

`ChatPage.tsx:361` — `setLiveReasoning(reasoningRef.current)` se llama en cada `reasoning_token`. Si el usuario tipea mientras el modelo está en streaming, ambos setStates compiten por renders.

### 2.4 Fixes priorizados

| # | Fix | Archivo | Impacto estimado |
|---|---|---|---|
| 1 | Envolver `LiminalAssistantMsg` en `React.memo` (con comparator sobre `blocks`, `reasoning`, `streaming`) | `LiminalAssistantMsg.tsx` | **30-40 %** menos renders |
| 2 | Envolver `LiminalMd` en `React.memo` | `LiminalMd.tsx:180` | 25-35 % en mensajes con markdown |
| 3 | Usar `useDeferredValue` o separar `inputDraft` state para no bloquear con los re-renders de `messages` | `ChatPage.tsx` (donde está `const [input, setInput] = useState(...)`) | Input se siente 2-3× más responsivo |
| 4 | Sacar el read de `scrollHeight` a un `requestAnimationFrame`, o usar CSS `field-sizing: content` (Chrome 123+) | `LiminalInput.tsx:45-50` | 5-10 ms por keystroke (chico pero compuesto con #1-2 suma) |
| 5 | Batch `setLiveReasoning` vía `useTransition` o throttle a 60 fps | `ChatPage.tsx:361` | 10-20 % cuando tipeás durante streaming |

**Nota sobre StrictMode**: es intencional en dev (detecta bugs), no recomiendo desactivarlo. El lag en producción no lo sufre 2×; lo que el usuario está viendo en dev es peor de lo real. Aun así, los fixes #1-3 son necesarios para producción.

### 2.5 Reproducción / verificación

1. DevTools → Profiler → Record.
2. Tipear en el textarea con ~10 mensajes históricos (uno con code block + markdown denso).
3. Antes: cada keystroke dispara render de `LiminalAssistantMsg × N`.
4. Después (con fix #1): sólo re-renderiza el draft en `LiminalInput`.
5. Target: <16 ms por keystroke (60 fps).

## 3. Tool calls — duplicación en la UI

**Síntoma**: cada invocación de tool aparece DOS VECES en el chat. Ej. `web_fetch` y `save_memory` que te mostraron.

### 3.1 Veredicto

**Híbrido: el backend emite `tool_start` dos veces + el frontend no deduplica por `tool_call_id`.**

### 3.2 Evidencia backend

El backend emite **dos eventos `tool_start`** para la misma invocación:

**Emisión #1** — durante streaming (`internal/agent/stream.go:96-102`)

```go
case provider.StreamEventToolCallStart:
    streamTelemetry(ctx, te, channelID, map[string]any{
        "type":         "tool_start",
        "name":         ev.ToolName,
        "tool_call_id": ev.ToolCallID,
    })
```

Ocurre cuando el provider (Anthropic, OpenRouter, Gemini) va mandando la tool call en streaming. **No incluye `input`** — todavía se está ensamblando.

**Emisión #2** — al arrancar la ejecución (`internal/agent/loop.go:340-346`)

```go
for _, tc := range resp.ToolCalls {
    if hasTelemetry {
        _ = telemetry.EmitTelemetry(ctx, msg.ChannelID, map[string]any{
            "type":         "tool_start",
            "name":         tc.Name,
            "input":        string(tc.Input),  // ← ahora sí con input
            "tool_call_id": tc.ID,
        })
    }
    // ... ejecuta la tool
}
```

Ocurre DESPUÉS del streaming, antes de ejecutar. Mismo `tool_call_id`, pero ahora SÍ trae el `input` completo.

**Orden real por invocación**:

```
tool_start      (id=X, sin input)     ← stream.go durante streaming
tool_assembled  (id=X)                ← stream.go al cerrar stream
tool_start      (id=X, con input)     ← loop.go antes de ejecutar
tool_done       (id=X, output)        ← loop.go tras ejecutar
```

**Respondiendo tu pregunta**: el agente NO ejecuta la tool dos veces. La ejecución ocurre una sola vez (`loop.go:381` → `executeWithRecover`). Lo que se duplica es el *evento* `tool_start`, no la *acción*.

### 3.3 Evidencia frontend

**Handler sin dedup** — `ChatPage.tsx:370-388`

```ts
case 'tool_start': {
  setMessages((prev) => [
    ...prev,
    bakeReasoningIfFirst({
      id: crypto.randomUUID(),              // ← nuevo ID cada vez
      role: 'tool',
      toolCall: {
        name: msg.name ?? '',
        input: msg.input ?? '',
        tool_call_id: msg.tool_call_id ?? '',
        done: false,
      },
      // ...
    }),
  ])
  break
}
```

Siempre hace `prev + [nueva entry]`. No verifica si ya existe un mensaje con ese `tool_call_id` y `done===false`.

**`tool_done` sólo actualiza el primero** — `ChatPage.tsx:390-410`

```ts
prev.map((m) => {
  const tc = m.toolCall
  if (tc && tc.tool_call_id === msg.tool_call_id && !tc.done) {
    return { ...m, toolCall: { ...tc, done: true, output: msg.output, ... } }
  }
  return m
})
```

Marca `done: true` SOLO en la primera coincidencia con `!tc.done`. El otro ChatMessage con el mismo `tool_call_id` queda en `done: false` para siempre → se ve como "tool que nunca terminó". Eso explica por qué el primero muestra sólo `1 lines 230ms` sin URL (venía del `tool_start` #1 sin input) y el segundo tiene URL completa (venía del `tool_start` #2).

### 3.4 Causa raíz

Dos eventos `tool_start` con el mismo `tool_call_id` — el frontend los trata como invocaciones independientes porque nunca se escribió la lógica de dedup/update.

### 3.5 Fix recomendado

**Opción preferida: dedup en frontend** (robusto, cubre cualquier provider).

En `ChatPage.tsx:370-388`, cambiar el handler `tool_start` a:

```ts
case 'tool_start': {
  setMessages((prev) => {
    const existingIdx = prev.findIndex(
      m => m.toolCall?.tool_call_id === msg.tool_call_id && !m.toolCall.done
    )
    if (existingIdx !== -1) {
      // Update existing — segundo tool_start sólo trae input, mergearlo.
      return prev.map((m, i) =>
        i === existingIdx
          ? { ...m, toolCall: { ...m.toolCall!, input: msg.input ?? m.toolCall!.input } }
          : m
      )
    }
    return [...prev, bakeReasoningIfFirst({ /* ... nueva entry ... */ })]
  })
  break
}
```

**Por qué frontend y no backend**: el backend tiene una razón legítima para la segunda emisión — no todos los providers streaman tool calls. Si el provider es no-streaming, `stream.go` nunca emite `tool_start`, y la única emisión llega en `loop.go`. Si limpiáramos ciegamente `loop.go`, los providers no-streaming dejarían de mostrar el inicio de la tool. Dedup en el cliente es idempotente y tolerante a ambos modos.

**Opción backend (alternativa)**: trackear en `loop.go` qué `tool_call_id`s ya se emitieron durante el streaming y saltarlos en la segunda emisión. Es más intrusivo pero ahorra el evento redundante en el wire.

## 4. Priorización y plan

### Alto impacto, bajo esfuerzo (hacer primero)

1. **Fix duplicación de tools** — 1 archivo, ~15 líneas. Elimina un bug cosmético visible en todas las conversaciones con tools.
2. **`React.memo` en `LiminalAssistantMsg` + `LiminalMd`** — 2 archivos, comparator manual. Resuelve el grueso del lag de tipeo.

### Medio esfuerzo, alto retorno

3. **Cablear Memory tab a `/api/memory` real** — requiere solo mapper cliente (derivar `confidence`/`kind`/`cluster-fallback` de los campos existentes). La pantalla queda operativa con data verdadera.
4. **`useDeferredValue` para el historial en `ChatPage`** — aísla el tipeo del re-render del historial.

### Alto esfuerzo, hacer cuando se quiera cerrar Memory al 100 %

5. **Migración backend: agregar `cluster` a `memory_entries` + clasificador en Curator.**
6. **Construir `/api/knowledge`**: migración con `access_count`/`last_accessed_at`/`summary`/`page_count`, endpoint REST, contador de injections.

### No hacer / low priority

- Desactivar `React.StrictMode` — el lag que sufre en dev no lo sufre prod; StrictMode detecta bugs reales y sacarlo es tirar señal.
- Rediseñar el formato de stream events del backend — la dedup en cliente es suficiente; cambiar el wire es over-engineering.
